# Article Check V4 敏捷迭代计划

> 配套 `docs/v4-architecture.md` 的实施手册

---

## 1. 目的

本文档用于将 V4 架构方案转化为可执行的敏捷实施计划，确保每个阶段都有：

- 清晰目标
- 明确 backlog
- 可度量指标
- 可验收结果
- 可演示成果

---

## 2. 迭代总原则

### 2.1 核心需求优先级

所有迭代优先围绕以下一级需求展开：

1. 本科毕业论文格式核查
2. 参考文献有效性验证
3. 批量检测
4. 单篇检测
5. 输出清晰、可追溯、可定位的问题报告

### 2.2 敏捷原则

- 先收敛主线，再扩展高级能力
- 先把证据链打通，再做复杂 Agent 化
- 先保证 CLI/Web/Chat 一致，再做工作台增强
- 先逻辑共享，再做底层 KV 共享
- 每一阶段都必须有回归验证与 demo

### 2.3 交付节奏

建议采用：

- 周级里程碑
- 双周或三周阶段性交付
- 每阶段结束做一次验收评审

---

## 3. Phase A：收敛主线（2 周）

### 3.1 阶段目标

把当前分散的执行主线收敛成统一 runtime，为后续 V4 执行核提供稳定底座。

### 3.2 用户价值

- 同一篇论文在 CLI、Web、Chat 下结果一致
- 批量和流式批量不再是两套并发逻辑
- 参考文献检查第一次真正进入主主线

### 3.3 Backlog

#### A-1 Runtime 收敛

- 新建 `build_runtime()`
- 抽离统一配置装配
- 抽离统一 worker 注册
- 抽离统一 tool 注册

#### A-2 多入口统一

- CLI 走 runtime
- Web 走 runtime
- Chat 走 runtime
- 兼容旧入口

#### A-3 批量调度统一

- `review_batch` 改由统一 pool 调度
- `review_batch_stream` 改由统一 pool 调度
- 保持先到先出特性

#### A-4 文献阶段落地

- 打通 `ReferenceWorker`
- 接入底层文献引擎
- 将 reference 结果纳入报告与评分

#### A-5 数据模型落地

- 引入 `ReviewIntent`
- 引入 `ReviewPlan`
- 引入 `EvidenceRecord`

### 3.4 交付物

- runtime 模块初版
- 统一批量调度模块
- reference worker 可用版
- 新数据模型定义
- Phase A 验证报告

### 3.5 验收标准

- CLI/Web/Chat 同一论文的主要问题项一致
- `ReferenceWorker` 非 stub
- evidence 能进入最终报告
- 批量与流式批量共用一个调度器

### 3.6 Demo 脚本

1. 上传单篇本科论文
2. CLI 审查
3. Web 审查
4. Chat 审查
5. 对比三者结果一致性
6. 展示 reference 风险项进入报告

### 3.7 风险

- 旧入口兼容性
- runtime 改造影响现有脚本

---

## 4. Phase B：引入 V4 执行核（3-4 周）

### 4.1 阶段目标

将主流程显式化为 DAG 工作流，并接入统一上下文治理与事件日志。

### 4.2 用户价值

- 审查流程可解释
- 节点状态可追踪
- 长任务可恢复
- 后续高并发优化有了统一载体

### 4.3 Backlog

#### B-1 TaskGraph 落地

- 定义节点类型
- 定义依赖边
- 定义阶段计划
- 绑定 worker

#### B-2 ContextCurator 全量接入

- 所有 LLM worker 使用同一上下文生成接口
- 上下文压缩决策可观测
- 支持恢复与摘要回放

#### B-3 ContextCacheBus 初版

- 逻辑缓存统一接口
- 支持 pack 读写
- 支持 acquire/release

#### B-4 事件日志与 checkpoint

- 记录节点生命周期事件
- 保存 checkpoint
- 支持恢复执行

### 4.4 交付物

- `orchestrator_v4` 初版
- TaskGraph 执行器
- ContextCurator 集成版
- ContextCacheBus 逻辑版
- 事件与 checkpoint 模块

### 4.5 验收标准

- DAG 可驱动主流程
- 节点状态可查询
- 长任务可从 checkpoint 恢复
- 所有 LLM worker 不再各自拼接上下文

### 4.6 Demo 脚本

1. 启动一次深度审查
2. 展示节点执行顺序
3. 中途终止
4. 从 checkpoint 恢复
5. 最终输出报告

### 4.7 风险

- 事件模型设计过重
- checkpoint 粒度不当导致实现复杂

---

## 5. Phase C：PolyKV / Fork 化（4-6 周）

### 5.1 阶段目标

把共享上下文从逻辑缓存升级为可复用的 pack 体系，并建立 fork/diff 和批次复用调度。

### 5.2 用户价值

- 批量同模板论文审查更快
- 高并发场景下 token 成本下降
- 多 worker 对同一论文的重复处理减少

### 5.3 Backlog

#### C-1 Pack 体系

- 模板包
- 投稿包
- 结构包
- 章节包
- 文献包

#### C-2 Fork/Diff 体系

- shared pack
- fork 引用
- diff 存储
- merge 策略

#### C-3 批次聚类

- 按模板聚类
- 按学院聚类
- 按期刊聚类
- 按学科主题聚类

#### C-4 reuse-aware 调度

- 关键路径优先
- 复用率优先
- 命中率统计
- token 节省统计

### 5.4 交付物

- pack 数据结构
- fork/diff 存储模块
- 聚类调度器
- 复用指标看板

### 5.5 验收标准

- 可观察到 pack 复用
- 可观察到 token 节省
- 同模板批量处理速度优于无聚类方案

### 5.6 Demo 脚本

1. 选择同学院同模板 20 篇论文
2. 展示批次聚类结果
3. 展示 pack 命中率
4. 展示吞吐与 token 节省结果

### 5.7 风险

- 过早耦合真实模型 KV
- 复用策略设计复杂度高

---

## 6. Phase D：领域强化（4 周）

### 6.1 阶段目标

围绕论文审改场景构建真正有壁垒的领域能力。

### 6.2 用户价值

- 不仅能查格式，还能查引用质量
- 不仅能查引用质量，还能查相关工作覆盖
- 不仅能发现问题，还能给修正闭环

### 6.3 Backlog

#### D-1 新颖性与遗漏引用子图

- coverage gap 分析
- 缺失关键文献推荐
- 相关工作覆盖摘要

#### D-2 自动修正闭环

- 生成修正计划
- patch draft
- patch apply
- recheck
- rollback

#### D-3 Web 工作台增强

- 证据面板
- 节点状态面板
- 人工改判
- 二次复审

### 6.4 交付物

- 文献覆盖分析结果
- 自动修正闭环能力
- Web 工作台增强版

### 6.5 验收标准

- 报告包含“格式 + 引用真实性 + 覆盖不足”三层结论
- 自动修正可回滚
- Web 端支持证据查看与改判

### 6.6 Demo 脚本

1. 上传一篇含格式问题和引用问题的论文
2. 展示 evidence
3. 展示遗漏引用候选
4. 应用自动修正
5. 重新审查

### 6.7 风险

- 文献分析可能引入外部 API 抖动
- 自动修正可能误改文档

---

## 7. Phase E：真实系统级优化（后续）

### 7.1 阶段目标

在实际本地 GPU 或推理服务部署场景中，推进更底层的性能与缓存优化。

### 7.2 范围

- 真实 KV 注入
- 分层缓存
- `RoPE-aware prefetch`
- `copy-on-write cache pages`
- 更接近 PolyKV/ForkKV 的底层接入

### 7.3 进入条件

- 主工作流稳定
- Pack 体系稳定
- cache bus 已经成为统一抽象
- metrics 体系成熟

### 7.4 验收方向

- 更低峰值内存
- 更高吞吐
- 更短时间到首结果
- 更高批量利用率

## 9. 建议的用户故事模板

### 格式核查类

- 作为学院管理员，我希望批量导入毕业论文并得到格式问题汇总，以便快速筛查不合规论文。
- 作为指导老师，我希望看到具体不符合模板的位置和原因，以便指导学生修改。

### 引用核查类

- 作为教师，我希望系统识别失效 DOI、可疑期刊和异常来源，以便发现虚假参考文献。
- 作为审查人员，我希望看到正文引用与参考文献条目之间的不一致，以便判断论文是否规范。

### 自动修正类

- 作为学生，我希望系统能自动修正常见格式问题，并允许我确认或回滚修改。

### 批量并行类

- 作为学院管理员，我希望在批量 20-100 篇论文时系统仍能稳定输出先到先出的结果。

## 10. 里程碑建议

### 里程碑 M1

Phase A 完成，主线统一，reference 进入主流程。

### 里程碑 M2

Phase B 完成，DAG + checkpoint + ContextCurator 全量接入。

### 里程碑 M3

Phase C 完成，pack/fork/reuse-aware 调度可观测。

### 里程碑 M4

Phase D 完成，形成面向论文审改的差异化能力。

### 里程碑 M5

Phase E 进入真实部署优化。

---

## 11. 结论

V4 的敏捷迭代必须围绕明确的领域目标推进：

- 本科毕业论文格式核查
- 参考文献有效性验证
- 批量与单篇兼容
- 报告清晰、证据可追溯、修正可闭环

只有在这个核心需求不漂移的前提下，PolyKV、Loop Engineering、TaskGraph、ContextCurator 等技术能力才真正有价值。
