# ArticleCheck 旧版与新版 UI/功能差异对比

## 1. 对比目的

本文档用于对比 `ArticleCheck-master` 旧版本与 `ArticleCheck-master-new` 新版本在页面 UI、交互流程和功能结构上的差异，便于在项目设计报告中说明版本演进逻辑。

说明：

- 旧版本路径：`d:/major/大三下/专业实训/ArticleCheck-master/ArticleCheck-master`
- 新版本路径：`d:/major/大三下/专业实训/ArticleCheck-master/ArticleCheck-master-new`

## 2. 总体定位差异

### 2.1 旧版本定位

旧版本更接近“研究型论文审查平台”，强调多场景扩展能力，除核心论文审查外，还提供：

- 文献检索
- 文献综述
- 投稿检查
- VSCode 插件联动
- 多入口统一 runtime

其设计目标是构建一个相对完整的论文审改与文献分析工作台。

### 2.2 新版本定位

新版本更接近“平台交付版最小实现”，聚焦于项目方平台上线需求，只保留最关键主线：

- WebDemo 前端
- FastAPI 网关
- 论文审查工作台
- 证据定位与报告问答
- Dify 服务接入
- 官方认证接入

新版本强调的是“平台化部署”和“最小交付闭环”，不再追求旧版本那种多业务页面并行展开的形态。

### 2.3 定位变化总结

从产品策略看，版本演进体现为：

- 从“功能扩展型研究系统”收束为“可上线的最小平台系统”
- 从“多模块综合学术工具”收束为“论文审查主线工作台”
- 从“强调功能广度”转向“强调部署稳定性与平台接入”

## 3. 路由与页面结构差异

### 3.1 旧版本页面结构

旧版本在前端路由中包含 5 个主要页面：

1. 控制台 `/`
2. 论文审查 `/review`
3. 文献检索 `/literature`
4. 文献综述 `/survey`
5. 投稿检查 `/submission`

对应代码见：

- [App.jsx](file:///d:/major/大三下/专业实训/ArticleCheck-master/ArticleCheck-master/article_check/web/frontend/src/App.jsx)

### 3.2 新版本页面结构

新版本前端路由收缩为 2 个页面：

1. 控制台 `/`
2. 论文审查 `/review`

对应代码见：

- [App.jsx](file:///d:/major/大三下/专业实训/ArticleCheck-master/ArticleCheck-master-new/article_check/web/frontend/src/App.jsx)

### 3.3 结构变化分析

新版删除了：

- 文献检索页
- 文献综述页
- 投稿检查页

这意味着新版不再把“论文写作辅助生态”作为前端重点，而是把所有入口收束到论文审查工作台。

## 4. 导航设计差异

### 4.1 旧版本导航

旧版本侧边栏包含五项导航：

- 控制台
- 论文审查
- 文献检索
- 文献综述
- 投稿检查

其导航表达的是“多业务模块并列”的产品结构。

### 4.2 新版本导航

新版本侧边栏仅保留：

- 控制台
- 论文审查

其导航表达的是“单主线聚焦”的平台结构。

### 4.3 导航层面总结

旧版本强调模块丰富度，新版本强调任务聚焦度。  
从 UI 信息架构角度看，新版本显著减少了一级导航数量，降低了用户选择成本，也更适合平台 WebDemo 交付。

## 5. 首页设计差异

### 5.1 共同点

两版首页都保留了以下结构：

- Hero 介绍区
- 系统状态面板
- 功能说明卡片
- 当前主线说明
- 推荐入口跳转

说明新版并没有推翻旧版的设计语言，而是在保留视觉骨架的前提下收缩内容。

### 5.2 旧版本首页特点

旧版本首页功能入口较多，首页承担多模块导航职责，入口卡片包括：

- 论文审查
- 文献检索
- 自动综述
- 投稿检查

系统状态区更偏向展示：

- DeepSeek 接入状态
- 检索源数量
- 模板数量

### 5.3 新版本首页特点

新版本首页仅保留一个主入口：

- 论文审查

状态区从旧版的“DeepSeek 已接入/未配置”转为新版的：

- Dify 已接入/待配置
- AI Provider
- 模板数量

### 5.4 首页变化总结

首页的核心变化不是视觉重构，而是信息重心变化：

- 旧版：强调多功能入口与本地审查能力
- 新版：强调平台化接入、Dify 联动和单主线交付

## 6. 论文审查页差异

### 6.1 共同点

旧版与新版的论文审查页整体结构高度一致，都保留：

- 上传区
- 审查控制按钮
- 深度审查开关
- 待审文件队列
- 报告工作台
- 证据定位
- 正式报告预览
- 报告问答

说明新版的核心 UI 资产基本沿用了旧版本已经成熟的“审查工作台”方案。

### 6.2 差异点一：业务角色变化

旧版本中的审查页更偏“研究型审改工作台”，与文献检索、综述、投稿检查组成一个更完整的学术辅助生态。

新版本中的审查页则变成“平台最小交付主界面”，几乎承载了全部核心业务价值，页面的重要性进一步提高。

### 6.3 差异点二：深度审查的品牌语义变化

旧版本顶部状态强调：

- `DeepSeek 已接入`
- `V4 Runtime`

新版本顶部状态强调：

- `Dify 已接入`
- `Platform Runtime`

这代表新版本在 UI 层面从“模型驱动”转向“平台工作流驱动”。

### 6.4 差异点三：工作台上下文变化

旧版本的工作台属于整个多功能系统中的一个页面；

新版本的工作台则是整个系统的核心交付界面，因此在设计报告中，新版可以被描述为：

> 以论文审查报告工作台为中心的单主线平台界面。

## 7. 接口能力差异

### 7.1 旧版本接口能力

旧版本前端接口包括：

- 状态查询
- 文件上传
- 单篇审查
- 深度审查
- 报告问答
- 证据片段
- 文献检索
- 文献综述
- 投稿检查
- 批量流式审查

### 7.2 新版本接口能力

新版本前端接口收缩为：

- 状态查询
- 文件上传
- 单篇审查
- 深度审查
- 报告问答
- 证据片段
- 报告文件访问
- 批量流式审查

### 7.3 接口层变化总结

删除的主要是外围辅助型能力：

- 文献检索
- 文献综述
- 投稿检查

保留的是核心审查闭环能力：

- 上传
- 检测
- 定位
- 报告
- 问答

## 8. 状态展示差异

### 8.1 旧版本状态表达

旧版本状态区重点强调：

- DeepSeek 是否配置
- 文献检索源数量
- 系统模板数量

说明旧版本更强调“本地审查能力 + 扩展检索能力”。

### 8.2 新版本状态表达

新版本状态区重点强调：

- Dify 是否接入
- AI Provider
- 平台 Runtime
- 模板数量

说明新版本更强调“网关接入能力 + 外部工作流能力 + 平台化部署形态”。

## 9. 视觉设计差异

### 9.1 一致性

两版前端视觉设计整体保持高度一致：

- 同样的左侧栏 + 顶栏 + 主工作区布局
- 同样的卡片化结构
- 同样的圆角按钮、胶囊标签和浅色背景
- 同样的正式报告化风格

### 9.2 差异性

新版的视觉差异主要体现在信息内容上，而非样式系统上：

- 视觉风格基本沿用旧版
- 交互框架沿用旧版
- 但页面内容大幅收缩，信息密度更低
- 一级入口更少，用户路径更短

因此可以认为：

> 新版本并不是一次大规模 UI 重做，而是在旧版成熟工作台设计基础上的业务收缩与交付聚焦。

## 10. 适合写入设计报告的总结

可以将旧版与新版差异概括为以下几点：

1. **产品定位变化**
   - 旧版是多功能研究型论文审查平台，新版是平台化最小交付版

2. **页面结构变化**
   - 旧版保留 5 个核心页面，新版收缩为 2 个页面

3. **功能边界变化**
   - 旧版覆盖文献检索、综述、投稿检查等外围模块，新版只保留论文审查主线

4. **状态表达变化**
   - 旧版强调 DeepSeek 与 V4 Runtime，新版强调 Dify 与 Platform Runtime

5. **交互策略变化**
   - 旧版强调多模块探索，新版强调单主线快速交付

6. **UI 演进方式**
   - 新版复用了旧版成熟的工作台设计语言，没有推翻重做，而是以最小改动实现平台场景迁移

## 11. 可直接引用的报告表述

> 与旧版本相比，新版本在前端页面结构上进行了明显收缩。旧版本采用多模块学术工作台模式，除论文审查外，还包含文献检索、文献综述和投稿检查等扩展页面；而新版本则以平台最小交付为目标，仅保留控制台和论文审查两个核心页面。虽然新版在视觉风格上延续了旧版的工作台式设计语言，但在功能组织上进一步强化了“论文审查主线”，并将系统状态表达从 DeepSeek 与 V4 Runtime 转向 Dify 与 Platform Runtime，体现出系统从研究型原型向平台化交付版本的演进趋势。
