# ArticleCheck 新版本页面 UI 交互设计说明

## 1. 文档目的

本文档面向 `ArticleCheck-master-new` 当前前端实现，用于梳理新版页面的 UI 结构、交互流程、样式语言与页面优化思路，便于直接写入项目设计报告中的“页面 UI 交互设计”部分。

项目路径：

- `d:/major/大三下/专业实训/ArticleCheck-master/ArticleCheck-master-new`

## 2. 新版本 UI 设计定位

新版前端不再延续旧版“多模块学术平台”的页面扩张方式，而是收束为围绕论文审查主线展开的最小交付版 WebDemo。其界面设计目标主要体现在以下四点：

- 以论文审查工作台作为唯一核心业务页面
- 以正式报告和问题定位作为结果呈现中心
- 以证据链、原文片段和报告问答增强可解释性
- 以简洁入口和清晰层次降低普通用户的学习成本

因此，新版本 UI 的核心不是展示尽可能多的系统能力，而是让用户以最短路径完成“上传论文、发起审查、查看结论、定位问题、安排修改、导出报告”的完整闭环。

## 4. 页面总体布局设计

### 4.1 总体布局形式

新版前端采用典型的工作台式三段布局：

1. 左侧导航栏
2. 顶部状态栏
3. 中部主工作区

这种布局既保留了平台型界面的稳定结构，又通过压缩页面数量和简化入口，让用户能够快速聚焦到论文审查主任务。

### 4.2 左侧导航栏设计

左侧导航仅保留两个一级入口：

- 控制台
- 论文审查

设计意义在于：

- 减少用户在首页的选择负担
- 明确系统主线只围绕论文审查展开
- 与“最小平台交付版”的产品定位保持一致

左侧下方仍保留“统一审查主线”说明卡片，用于强调系统的三项核心价值：

- 格式预警与定位
- 文献有效性核验
- 审改建议与正式报告

### 4.3 顶部状态栏设计

顶部栏主要用于展示：

- 当前系统标题
- Dify 接入状态
- Platform Runtime 标识
- GitHub 外链入口

与旧版相比，顶部设计不再突出模型品牌，而是更强调平台接入状态和运行环境，这说明新版已从“模型功能演示”转向“可交付平台界面”。

## 5. 首页 UI 交互设计

对应文件：

- `src/pages/Dashboard.jsx`

### 5.1 页面定位

首页承担的是“说明型门户”角色，主要负责：

- 向用户说明当前版本的系统定位
- 展示平台运行状态和接入信息
- 提供进入论文审查页面的主入口
- 对核心能力和业务主线进行概括说明

### 5.2 页面组成

当前首页主要由以下四个部分构成：

1. Hero 介绍区
2. 系统状态面板
3. 设计目标说明区
4. 当前主线说明区

需要特别说明的是，前期版本中首页曾存在重复的功能入口卡片和“推荐入口”板块，但在当前版本中已删除，仅保留 Hero 区中的“进入报告工作台”按钮和侧边栏导航入口，以避免重复跳转和视觉干扰。

### 5.3 交互设计

#### Hero 区

- 展示系统定位文本
- 提供“进入报告工作台”按钮
- 强调当前版本的技术组合为 `WebDemo + FastAPI 网关 + Dify 多应用编排 + 官方认证接入`

设计作用：

- 帮助用户在进入首页后快速建立系统认知
- 将用户直接引导到最重要的论文审查场景

#### 状态面板

展示内容包括：

- 系统状态
- Dify 接入状态
- 模板数量
- AI Provider

交互特点：

- 以状态展示为主，不增加多余操作
- 使用胶囊标签增强可视化识别

#### 设计目标说明区

该区域通过卡片化方式概括六类能力：

- 正式报告模板
- 证据定位
- 文献风险分析
- 批量并行审查
- 报告问答
- 导出与归档

设计意义：

- 用简洁模块说明核心能力，而不是用大量跳转入口打散注意力
- 让首页承担“介绍系统价值”的作用，而不是“堆叠功能按钮”的作用

#### 当前主线说明区

该区域用时间线式卡片说明论文审查的主线流程，包括：

- 上传论文后生成统一结构化报告
- 格式、文献、内容问题汇入同一报告
- 证据链与问答共用统一数据模型
- 正式审改报告作为核心交付物输出

这一区域强化了系统的流程感和闭环感，便于在设计报告中说明页面如何服务于业务主线。

### 5.4 适合报告中的描述

> 新版首页采用说明型门户设计，不再堆叠重复功能入口，而是通过系统定位说明、平台状态展示、设计目标卡片和主线流程说明，帮助用户快速理解平台价值，并通过单一主按钮直接进入论文审查工作台。

## 6. 论文审查页 UI 交互设计

对应文件：

- `src/pages/ReviewPage.jsx`
- `src/components/ReviewStudio.jsx`

### 6.1 页面定位

论文审查页是新版系统最核心的业务页面。其设计目标不是简单返回一份检测结果，而是构建一个面向用户的“论文审查报告工作台”，使用户能够在同一页面内完成上传、审查、查看、定位、追问和导出的全流程操作。

### 6.2 页面结构

当前页面可划分为两大层次：

1. 上层审查控制区
2. 下层报告工作台区

这种“先操作、后消费结果”的布局符合用户任务流，能够减少界面切换和认知跳跃。

### 6.3 上层审查控制区设计

主要元素包括：

- 上传论文按钮
- 正式审查按钮
- 流式批量按钮
- “启用深度内容审查”开关
- 待审文件列表
- 当前模式状态标签

#### 上传交互

- 用户点击“上传论文”后调用本地文件选择器
- 支持上传 `tex`、`docx`、`doc`、`pdf` 等常见论文文件
- 上传成功后文件进入待审列表
- 通过去重逻辑避免重复文件反复加入队列

#### 单篇审查交互

- 点击“正式审查”后逐篇调用审查接口
- 页面进入加载状态
- 审查完成后将结构化结果写入报告队列
- 首篇结果会自动被选中，便于用户直接查看

#### 批量流式交互

- 点击“流式批量”后，通过 SSE 持续接收结果
- 每完成一篇论文，即动态加入报告队列
- 用户无需等待所有任务完成即可提前查看已返回结果

#### 深度审查交互

- 通过复选框控制是否启用深度内容审查
- 开关状态会同步反映到上方状态标签
- 当前文案已统一为“启用深度内容审查”，避免出现不必要的模型品牌信息

### 6.4 报告工作台设计

`ReviewStudio.jsx` 是新版 UI 的核心结果组件。当前版本已经从偏开发视角的展示方式，收束为更适合普通用户理解和截图展示的结果工作台。

当前工作台主要由以下区域组成：

1. 审查报告总览区
2. 打印版报告预览区
3. 报告队列区
4. 定位导航区
5. 格式预警与定位
6. 文献预警与定位
7. 报告片段与证据链
8. 原文片段联动预览
9. 审改行动清单
10. 内容审查摘录
11. 报告问答区

需要说明的是，当前版本已删除以下不再面向用户的内容：

- 报告模板说明板块
- 重复的打开/打印按钮
- 统一详情视图
- 工作流执行轨迹

删除这些区域后，页面信息层级更集中，普通用户不再需要先理解系统内部结构，便可以直接聚焦问题和修改建议。

#### 6.4.1 审查报告总览区

展示内容包括：

- 报告标题
- 任务编号
- 综合评分
- 风险结论
- 统计指标卡片

设计特点：

- 采用“正式报告封面”式的视觉组织方式
- 用标题、评分和风险描述形成页面第一视觉焦点
- 适合直接截图用于项目展示和报告配图

#### 6.4.2 打印版报告预览区

该区域提供两层能力：

- 通过上方按钮打开打印版报告或直接打印 / 导出 PDF
- 在页面内部嵌入正式 HTML 报告预览

相比早期版本，当前页面只保留一组主操作按钮，避免同一功能在多个板块中重复出现。

#### 6.4.3 报告队列区

该区域用于展示：

- 已完成审查的报告列表
- 当前选中的报告项
- 每份报告的标题、评分和问题数量

交互逻辑：

- 点击报告项即可切换当前查看对象
- 支持单篇审查与批量流式结果的统一管理

#### 6.4.4 定位导航区

导航区优先呈现：

- 高风险证据
- 可定位的问题条目
- 快速跳转入口

设计意义：

- 让用户可以从长报告中直接跳到最关键的问题片段
- 降低手动查找的时间成本

#### 6.4.5 格式预警与定位

该区域集中展示格式层面的风险问题，包括：

- 模板不一致
- 标题或图表格式错误
- 章节缺失
- 版式异常

每条问题均包含：

- 严重度
- 问题描述
- 定位信息
- 修改建议
- 定位操作

当前版本已经对该板块的表格宽度和布局进行了优化：

- 将问题表格改为固定列宽的稳定结构
- 外层卡片增加 `min-w-0` 约束
- 问题区改为单列纵向排列
- 内容单元格启用自动换行

这样可以避免板块宽度溢出，使其与其他卡片边界保持一致。

#### 6.4.6 文献预警与定位

该区域聚焦参考文献相关风险，包括：

- 正文引用与参考文献列表不一致
- 参考文献缺失
- DOI 缺失
- 可验证性不足

它与“格式预警与定位”保持相同的结构和视觉规则，从而保证页面在纵向浏览时具有一致性，也便于用户按类别理解问题来源。

#### 6.4.7 报告片段与证据链

该区域用于展示关键问题对应的证据摘要，包括：

- 问题严重度
- 所属阶段
- 问题位置
- 问题摘要
- 修改建议

用户点击“跳转到对应片段”后，可直接联动到页面内对应证据块，强化问题与证据之间的对应关系。

#### 6.4.8 原文片段联动预览

当用户选中某条证据后，页面会自动请求对应的源文片段，并在此区域展示：

- 原文来源名称
- 问题摘要
- 定位摘要
- 焦点行
- 原文上下文内容

这一设计使界面不仅告诉用户“哪里有问题”，还能够展示“为什么判定这里有问题”，增强系统可信度。

#### 6.4.9 审改行动清单

该区域根据严重度输出修订优先级，帮助用户明确：

- 先修复哪些问题
- 哪些问题会直接影响提交
- 哪些问题适合作为第二阶段修订内容

它将分散的检测结果转换为更接近实际写作流程的行动建议，是用户视角优化中的重要部分。

#### 6.4.10 内容审查摘录

该区域用于保留最影响论文质量的内容问题，如：

- 结构表达不清
- 论证薄弱
- 写作问题集中段落

它不追求展示全部内容问题，而是提供有限且高价值的摘录，避免信息过载。

#### 6.4.11 报告问答区

用户可以围绕当前结构化报告继续提问，例如：

- 我应该先修改哪几个问题？
- 哪类问题最影响提交质量？
- 哪些证据说明当前风险最高？

设计意义：

- 让报告从静态结果升级为可追问结果
- 提高结果解释效率
- 增强系统的人机协同特征

### 6.5 模板兜底设计

新版 `ReviewStudio.jsx` 内置 `DEMO_REPORT` 模板数据，即使没有真实审查结果，页面仍可完整渲染。

其主要优势在于：

- 便于前端开发阶段做页面验证
- 便于系统演示与截图
- 保证在无数据状态下界面仍保持完整和稳定

## 7. 样式系统设计

对应文件：

- `src/index.css`

### 7.1 基础视觉风格

新版样式系统延续工作台风格，但在当前版本中更强调“正式报告感”和“卡片边界一致性”。其基础视觉特征包括：

- 浅暖色背景
- 深色侧边栏
- 白色半透明卡片
- 柔和阴影与大圆角
- 统一的标题层级与说明文本节奏

### 7.2 关键样式组件

样式系统中大量使用统一类名，包括：

- `.btn-primary`
- `.btn-outline`
- `.capsule`
- `.surface-card`
- `.upload-panel`
- `.report-hero`
- `.print-preview-frame`
- `.evidence-card`
- `.answer-panel`

这使页面能够在多个内容区之间维持统一的边框、圆角、阴影和间距逻辑。

### 7.3 细节优化点

当前版本在视觉细节上进行了以下优化：

- 统一审查页核心文案的中文表达
- 删除不必要的模型品牌信息
- 调整控制台与卡片内部间距
- 统一问题区块的边界和列宽
- 让按钮、问题表格、证据卡片之间的留白更加均衡

这些细节优化使页面整体更偏向用户视角，而不再像开发调试面板。

## 8. 新版交互设计总结

新版页面 UI 交互设计的核心特征可以总结为：

1. 单主线聚焦  
   系统仅围绕“控制台 + 论文审查”两页组织交互，避免旧版多入口造成的分散感。

2. 工作台化整合  
   审查页将上传、执行、查看、定位、问答、打印整合在同一页面，形成闭环。

3. 用户视角收束  
   删除工作流执行轨迹、统一详情视图等偏内部的展示内容，使界面更适合普通用户使用。

4. 报告导向呈现  
   结果不再以调试数据堆叠为主，而是转化为正式报告、问题定位和行动清单。

5. 证据驱动解释  
   通过证据链、源文片段联动和报告问答提升可解释性与可信度。

6. 边界与版式稳定  
   对“格式预警与定位”和“文献预警与定位”做了布局重排和边界收口，使其与其他板块保持统一对齐。

## 9. 可直接写入报告的表述

> `ArticleCheck-master-new` 的页面 UI 设计在整体上采用了工作台式布局，但在产品结构上明显收束为面向论文审查主线的最小交付版。系统仅保留控制台和论文审查两个核心页面，通过左侧简化导航、顶部平台状态栏和中心报告工作台构建统一交互框架。其中，论文审查页采用“审查控制区 + 报告工作台区”的双层结构，将论文上传、单篇与批量审查、问题定位、正式报告预览、原文证据联动和报告问答整合在同一页面中。界面优化过程中删除了重复入口、冗余按钮以及偏开发视角的工作流展示，使页面更加符合普通用户的阅读和操作逻辑。视觉设计上，系统使用卡片化布局、大圆角、胶囊状态标签和正式报告式的信息组织方式，在保证专业性与可信度的同时，也增强了界面的清晰度、美观性和截图展示效果。
